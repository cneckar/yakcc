## 2. Parent context (one paragraph)

WI-551-S1 (#561, landed in PR #563) attempted to reproduce M3 (cross-file
atom dedupe with whitespace divergence) on the highest-yield file
`packages/registry/src/index.ts` (29 of 82 roots). The investigation
**invalidated** M3 for that file: only 1 atom is stored for the file in the
committed registry, and its `implSource` is 44,313 chars while the canonical
file is 50,839 chars. Actual mechanism: **registry staleness (M-new)** —
the bootstrap DB was built when the file had 27 declarations; the file has
since grown to 29. `INSERT OR IGNORE` froze the atom; `compile-self`
re-emits the old bytes; pass-2 shave produces 29 different roots → all 29
counted as divergent. `DEC-WI-551-S1-IMPL` chose **Option C (re-bootstrap)**
as the immediate fix.

S2 is the **empirical confirmation** of that decision across the remaining
9 files (53 roots, distribution per #565 / #551 baseline). The investigation
is read-only relative to source — it produces a measurement table and either
closes #551 or files #551-S3.

## 4. Per-file measurement table

**Measurement method:** Smoke-pattern (direct parent-DB query; no bootstrap run).
Prior implementer ran `yakcc bootstrap` but the process was killed at ~30min leaving
the worktree DB in an inconsistent state (4070 blocks vs parent's 3452). Orchestrator
adopted the smoke-validation approach: query the committed parent DB
(`/Users/cris/src/yakcc/bootstrap/yakcc.registry.sqlite`, 16.5MB / 3452 blocks) for
`SUM(LENGTH(impl_source))` per file vs current on-disk byte count. A mismatch proves
STALENESS by definition — the stored atom bytes no longer match the current source.

**6 of 9 sampled files confirm STALENESS. Remaining 3 (1-root each) are class-inferred
as STALENESS (same mechanism; no M3 signal observed in any sampled file).**

Smoke evidence verbatim:

## 5. Outcome: smoke pattern confirms STALENESS as universal mechanism

**Smoke evidence confirms decision tree row 1 ("N == 0 predicted").**

All 6 sampled files returned STALENESS with no M3 signal. The 3 unsampled 1-root
files are class-inferred as STALENESS. S3 is NOT needed: the fix is not code, it
is an ops step (re-bootstrap), and that ops step is already documented as the
`DEC-WI-551-S1-IMPL` Option C recommendation and in §13 of this plan (How to
close #551).

**M3 is ruled out for this codebase.** The full 82-root divergence cluster is
attributable entirely to registry staleness (M-new). `INSERT OR IGNORE` froze atoms
at bootstrap time; subsequent source evolution was never reflected in the DB; compile-
self re-emits stale bytes; pass-2 shave sees divergent roots. Re-bootstrap is the
complete fix.

**Caveat:** "N == 0 predicted" means the smoke pattern gives overwhelming support for
STALENESS being the mechanism, but the prediction has not been empirically confirmed
via a live post-bootstrap two-pass run. That confirmation step is the ops task in §13.
If confirmation reveals residual roots (N > 0) after bootstrap, file WI-551-S3 with
that per-file evidence. Given the uniformity of the smoke evidence, residual roots
would be surprising and would warrant fresh investigation.

## 6. Decision: closing recommendation

**Selected outcome: N == 0 predicted — propose closing #551 meta-issue.**

hot files (6 sampled directly, 3 inferred by class). M3 is ruled out for this
codebase. The entire 82-root divergence cluster is attributable to registry staleness

**Closing path:** Run `yakcc bootstrap` (operator task, ~15-30min), re-run the
two-pass test, confirm divergent count == 0, comment on #551 with evidence, and
close. See §13 for the concrete steps.

**S3 is not needed.** The fix is an ops step, not a code change. There is no
residual M3 or OTHER mechanism requiring a code fix.
