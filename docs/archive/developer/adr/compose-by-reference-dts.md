# ADR: .d.ts Generation from SpecYak for Compose-by-Reference Type Safety

**ID:** DEC-COMPOSE-BY-REF-DTS-001
**Status:** ACCEPTED
**Date:** 2026-06-01
**Author:** implementer (wi-1046)
**Closes:** #1046
**Related:** DEC-COMPOSE-BY-REF-MANIFEST-001, DEC-COMPOSE-BY-REF-BUILD-001, DEC-V3-DISCOVERY-D4-001, #1043 (epic), #1044, #1045, #1047

---

## Context

`yakcc build` (#1045) materializes `.yakcc/atoms/<alias>.ts` for each referenced atom —
the runnable TypeScript implementation. However, reference source (code that imports from
`.yakcc/atoms/<alias>`) needs to _typecheck_ BEFORE `yakcc build` has been run. In a
TypeScript project, that means a `.d.ts` declaration file must exist at
`.yakcc/atoms/<alias>.d.ts` before any `.ts` implementation is present.

Without this, any project that imports a compose-by-reference atom fails `tsc --noEmit`
until after the first build — making it impossible to typecheck during development
(IDE, CI lint gate) in the natural workflow where reference import lines are added before
the build runs.

---

## Decision

### The .d.ts is derived from the atom SpecYak, not from the implementation

The declaration file is generated entirely from the atom's `SpecYak` signature: the
`inputs` array (parameter names and types) and the `outputs` array (return type).
It does NOT require the `.ts` implementation to exist. This is the key property:

```
reference source imports atom → .d.ts present → tsc finds zero errors
                                  (no .ts impl needed)
```

The `.ts` impl materializes later via `yakcc build` (#1045). The `.d.ts` is the "typed
half" that makes pre-build typechecking possible.

### generateAtomDts() — single authority for .d.ts text

`packages/compile/src/atom-dts.ts` exports `generateAtomDts(spec: SpecYak, symbol: string): string`.
It is the sole authority for the `.d.ts` content. No other code derives declaration text
from a SpecYak. The generated text has the form:

```typescript
// This file is generated by yakcc from the atom SpecYak signature (#1046).
// DEC-COMPOSE-BY-REF-DTS-001 — type-only; no runtime value is present here.
// `yakcc build` materialises the real implementation as the sibling .ts file.
export declare function <symbol>(<params>): <returnType>;
```

Return-type mapping (simplest faithful representation):

| Outputs | TypeScript return type |
|---------|----------------------|
| 0       | `void`               |
| 1       | that output's `type` string (e.g. `string`) |
| N       | positional tuple `[t0, t1, …]` |

Parameter name synthesis: if an input has an empty or absent `name`, the generator
synthesises `arg0`, `arg1`, … positionally. This prevents syntactically invalid output
for atoms whose spec omits parameter names.

Failure is loud: `generateAtomDts` throws a descriptive `Error` if `spec.inputs` or
`spec.outputs` is not an array, or if `symbol` is empty. No silent defaults.

### materializedDtsPath() — single authority for the output path

`packages/compile/src/project-manifest.ts` exports `materializedDtsPath(alias): string`
which returns `.yakcc/atoms/<alias>.d.ts`. This is the sole authority for the path shape.
`yakcc build` uses this function; no other path derivation is re-implemented elsewhere.

### yakcc build emits the .d.ts alongside the .ts

After writing `.yakcc/atoms/<alias>.ts` (via `assemble()`), the build command:

1. Calls `registry.getBlock(ref.root)` — loud failure on `null` (unexpected, since
   `assemble()` already resolved the root successfully).
2. Decodes `blockRow.specCanonicalBytes` via `JSON.parse(new TextDecoder().decode(...))`.
   This is the canonical round-trip used elsewhere in the codebase
   (e.g. `discovery-eval-full-corpus.test.ts`). Loud failure on decode error.
3. Calls `generateAtomDts(spec, ref.symbol)` where `ref.symbol` is the bound export name
   from the project manifest (e.g. `"asciiChar"`) — NOT the `spec.name` field.
4. Writes the result to `materializedDtsPath(ref.alias)` relative to project root.

The `.d.ts` and `.ts` share the same directory; `mkdirSync` is idempotent.

### Why the symbol comes from the manifest, not the spec

The spec's `name` field (e.g. `"ascii-char"`) is the atom's canonical name. The manifest
`symbol` (e.g. `"asciiChar"`) is the bound export name chosen by the project. These may
differ. The in-source import line `import { asciiChar } from ".yakcc/atoms/..."` must
use the manifest symbol, not the spec name. The `.d.ts` must declare exactly the symbol
that the source imports — so it uses `ref.symbol`.

### Consumed by #1047 (yakcc_reference MCP tool)

`generateAtomDts` is exported from the `@yakcc/compile` barrel so that the MCP tool
(#1047) can produce the `.d.ts` without re-implementing the spec-to-declaration logic.
This is the canonical integration point per the `@yakcc/compile` barrel comment.

---

## Alternatives considered

### Derive the .d.ts by emitting from the full impl

Using `tsc --declaration --emitDeclarationOnly` on the assembled `.ts` would produce a
correct `.d.ts`, but it requires the `.ts` to exist first — defeating the pre-build
typechecking goal. It also introduces a build-tool dependency and non-determinism if the
compiler changes its output format. Rejected.

### Ship no .d.ts; use `@ts-ignore` or ambient declarations

Requiring callers to add `@ts-ignore` on every reference import is fragile and defeats
the point of using TypeScript. Ambient module declarations (`declare module ".yakcc/atoms/*"`)
would be overly permissive (accepting any import regardless of shape). Rejected.

### One .d.ts per atom (not per alias)

A single `.d.ts` for all atoms would require a different module specifier strategy. The
alias-per-reference design (DEC-COMPOSE-BY-REF-MANIFEST-001) means each alias maps to an
independent file pair (`.ts` + `.d.ts`). This is consistent, simple, and file-system safe.

---

## Downstream consumers

- **`yakcc build` (#1045 / #1046)**: emits `.yakcc/atoms/<alias>.d.ts` alongside `.ts`
  for every reference in the project manifest.
- **`yakcc_reference` MCP tool (#1047)**: may call `generateAtomDts` to surface the type
  signature to the model as part of the reference handoff.

---

## Consequences

**Positive:**
- Reference source typechecks with ONLY the `.d.ts` present — no full build required.
- The `.d.ts` is deterministic and idempotent: same manifest + same registry →
  byte-identical output across runs.
- Failure is loud on missing block, malformed spec, or empty symbol — no hidden state
  corruption on partial builds.
- `generateAtomDts` is independently unit-tested (EC-1 through EC-6 in `atom-dts.test.ts`)
  and the build integration is tested end-to-end including the no-impl typecheck proof
  (`build.test.ts` EC-DTS-1, EC-DTS-2, EC-DTS-3).

**Constraints:**
- The SpecYak `inputs`/`outputs` schema must remain stable for the `.d.ts` to remain
  valid. A schema change (e.g. adding optional fields) requires a coordinated update to
  `generateAtomDts`.
- The tuple return type for multi-output atoms is a simplification. If a future spec
  version adds named-output conventions, `generateAtomDts` should produce named object
  types instead — but that is deferred until the spec has a canonical naming rule.

---

## Rollback boundary

This ADR covers:
- `packages/compile/src/atom-dts.ts` — `generateAtomDts` implementation
- `packages/compile/src/atom-dts.test.ts` — unit tests (EC-1 through EC-6)
- `packages/compile/src/index.ts` — barrel export of `generateAtomDts`
- `packages/cli/src/commands/build.ts` — `.d.ts` emission wiring
- `packages/cli/src/commands/build.test.ts` — integration tests (EC-DTS-1, EC-DTS-2, EC-DTS-3)
- This ADR file

`assemble.ts`, `project-manifest.ts`, and `manifest.ts` are untouched.
Reverting #1046 removes the `.d.ts` emission; reference source will fail `tsc --noEmit`
until `yakcc build` is run (restoring the #1045 baseline state).
